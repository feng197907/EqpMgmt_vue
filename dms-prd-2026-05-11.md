## 1. 产品概述

### 1.1 产品名称

**设备档案文档管理系统**（Device Document Management System，简称 DMS）

### 1.2 产品背景

在制药行业，设备档案管理是GMP合规的核心环节。设备从采购、安装、验证到退役的全生命周期中，会产生大量文档：URS（用户需求规格）、DQ（设计确认）、IQ（安装确认）、OQ（运行确认）、PQ（性能确认）、校准记录、维护记录、偏差报告、变更记录等。

传统管理方式存在以下痛点：
- 纸质文档查找困难，易丢失或损坏
- 多人同时需要同一份文档时协调成本高
- 版本混乱，无法追溯历史版本
- 审计准备耗时，容易遗漏文件
- 无法有效关联设备与文档

### 1.3 产品目标

构建一套符合中国GMP、FDA 21 CFR Part 11、EU GMP Annex 11要求的设备档案文档管理系统，实现：
- 设备档案全生命周期数字化管理
- 文档版本控制与审计追踪
- 设备-文档结构化关联
- 线上线下一体化借阅
- 合规风险预警

## 2. 产品定位与价值主张

### 2.1 市场定位

| 维度 | 定位 |
|------|------|
| 目标市场 | 中国中型制药企业（200-2000人，年营收1-50亿） |
| 设备规模 | 100-500台设备 |
| 文档规模 | 12,000-30,000份文档 |
| 价格区间 | 模块化SaaS定价：基础版5-10万/年，专业版15-30万/年，企业版30-60万/年 |

### 2.2 核心价值主张

**"设备档案全生命周期管理专家"**

1. **核心差异化**：国内首款专注于设备档案全生命周期管理的DMS系统
2. **设备-文档强关联**：建立设备与相关文档的结构化关联，告别文档孤岛
3. **线上线下一体化**：支持电子文档和纸质档案的统一管理
4. **NMPA原生合规**：内置中国GMP合规检查逻辑，开箱即用
5. **高性价比**：专为中型药企设计，20-50万区间无强力竞争者

### 2.3 竞争优势分析

| 竞品 | 优势 | 不足 | DMS机会 |
|------|------|------|---------|
| Veeva/MasterControl | 全球品牌，功能全面 | 价格昂贵，本土化不足 | 本土化+性价比 |
| 辛格迪翱泰 | 纸质档案管理 | 设备档案管理薄弱 | 设备档案强化 |
| 通用DMS(Alfresco) | 开源可定制 | 需大量定制开发 | 行业深度+开箱即用 |
| 内部自研 | 完全定制 | 维护成本高，迭代慢 | 专业产品持续迭代 |

## 3. 目标客户画像

### 3.1 核心用户角色

| 角色 | 职责描述 | IT水平 | 使用频率 |
|------|----------|--------|----------|
| **QA经理** | 审批变更/偏差报告，主导GMP审计 | 中等 | 高 |
| **设备工程师** | 日常巡检、维护保养、故障维修 | 偏低 | 高 |
| **验证工程师** | 编写DQ/IQ/OQ/PQ方案，执行验证活动 | 中等 | 中 |
| **档案管理员** | 文档分类归档、版本管理、借阅登记 | 偏低 | 高 |
| **生产主管** | 设备使用安排、监控运行状态 | 中等 | 中 |
| **计量工程师** | 制定校准计划、管理校准记录 | 中等 | 中 |

### 3.2 典型用户痛点

**QA经理痛点**：
- 审计时临时查找文档，手忙脚乱
- 担心文档版本过期未被发现
- 难以实时掌握全厂设备合规状态

**设备工程师痛点**：
- 需要知道某台设备的所有历史维护记录
- 设备说明书找不到或版本不对
- 维修后忘记上传记录

**档案管理员痛点**：
- 纸质档案借出后无法追踪
- 多人同时需要同一份文件
- 版本更新后无法回收旧版本

## 5. 功能需求详细说明

### 5.1 设备管理模块

#### 5.1.1 设备档案CRUD

**功能描述**：
建立设备主档案，记录设备基础信息和全生命周期状态。

**字段设计**：
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 设备编码 | 字符串 | 是 | 唯一编码，支持自定义规则 |
| 设备名称 | 字符串 | 是 | - |
| 规格型号 | 字符串 | 否 | - |
| 设备类别 | 枚举 | 是 | 生产设备/检验设备/公用工程等 |
| 制造厂商 | 字符串 | 否 | - |
| 出厂编号 | 字符串 | 否 | - |
| 采购日期 | 日期 | 否 | - |
| 安装位置 | 字符串 | 是 | 区域/楼层/房间 |
| 当前状态 | 枚举 | 是 | 调试/运行/待机/维修/停用/报废 |
| 关联组织 | 组织ID | 是 | 所属部门/车间 |
| 备注 | 文本 | 否 | - |

**业务规则**：
- 设备编码唯一性校验
- 状态变更需填写变更原因
- 停用/报废需走变更审批流程

**验收标准**：
1. [ ] 可创建设备档案，保存后自动生成设备ID
2. [ ] 可编辑设备信息，变更记录自动记入审计日志
3. [ ] 可软删除设备（状态变为"停用"），不影响历史数据
4. [ ] 设备列表支持按名称/编码/状态/部门筛选
5. [ ] 设备详情页展示关联的所有文档

#### 5.1.2 设备-文档结构化关联

**功能描述**（核心差异化）：
建立设备与文档的强关联，支持按文档类型分组展示。

**关联规则**：
| 文档类型 | 生命周期阶段 | 关联说明 |
|----------|-------------|----------|
| URS | 采购前 | 用户需求规格书 |
| DQ | 设计确认 | 设计确认报告 |
| IQ | 安装确认 | 安装确认方案+报告 |
| OQ | 运行确认 | 运行确认方案+报告 |
| PQ | 性能确认 | 性能确认方案+报告 |
| 设备说明书 | 全周期 | 设备随机资料 |
| 维护记录 | 运行中 | 保养、维修记录 |
| 校准记录 | 运行中 | 计量校准记录 |
| 变更记录 | 全周期 | 任何变更的记录 |
| 偏差报告 | 全周期 | 偏差调查记录 |
| 图纸 | 全周期 | CAD图纸、布置图 |
| 培训记录 | 全周期 | 操作培训记录 |

**业务规则**：
- 每台设备必须关联至少一个URS或设备说明书
- 验证文档（DQ/IQ/OQ/PQ）按生命周期阶段自动归档
- 设备状态变更时提示检查相关文档

**验收标准**：
1. [ ] 设备详情页按文档类型分组展示关联文档
2. [ ] 新增文档时可选择关联设备（支持搜索）
3. [ ] 设备停用时检查是否有关联文档未归档
4. [ ] 支持按设备查看文档变更历史

### 5.1.3 设备生命周期与状态变更

**功能描述**：
管理设备从入厂、安装、验证、运行、维护到退役的完整生命周期，记录每次状态变更的原因、责任人与时间戳，并关联相关文档（如IQ/OQ/PQ、维护记录、校准证书等）。

**关键字段/状态**：
- 状态：未交付 / 入厂 / 安装 / 验证中 / 运行 / 停机 / 维修 / 停用 / 报废
- 状态变更记录：变更人、变更时间、变更原因、关联文档ID列表

**业务规则**：
- 关键状态（如“停用”、“报废”）必须走变更审批流程并记录变更原因。
- 从“验证中”到“运行”需确认验证文档（IQ/OQ/PQ）状态为“生效”。
- 状态变更触发通知给设备管理负责人与档案管理员。

**验收标准**：
1. [ ] 可为每台设备记录并查看完整的状态变更历史
2. [ ] 关键状态变更必须填写变更原因并触发审批（如适用）
3. [ ] 状态变更事件可关联相关文档并在设备详情中一键跳转查看

### 5.1.4 设备台账与展示

**功能描述**：
设备台账提供分页、筛选与导出功能，展示设备关键属性、当前状态、所在部门和最近一次维护/校准时间，支持按部门、状态、型号、关键字等组合筛选。

**展示要点**：
- 列：设备编码、设备名称、规格型号、设备类别、当前位置、当前状态、最后维护时间、最后校准时间、操作（查看/编辑/导出）
- 列可自定义显示顺序和导出列集合

**验收标准**：
1. [ ] 台账支持分页与至少4种以上的组合筛选条件
2. [ ] 台账支持 CSV/Excel 导出（可选列）
3. [ ] 列表项支持快捷查看设备详情与进入设备审计/文档列表

### 5.1.5 设备接口与数据同步

**功能描述**：
为后续系统集成预留 RESTful API 接口，包括设备查询、设备详情、设备状态变更、设备文档列表、设备台账导出等接口；支持增量同步和基于时间戳的查询。

**接口示例**：
- `GET /api/devices` — 支持分页/筛选/字段选择
- `GET /api/devices/{id}` — 返回设备详情及关联文档元数据
- `POST /api/devices/{id}/status` — 提交状态变更（需权限）
- `GET /api/devices/{id}/documents` — 返回该设备关联文档列表

**安全与一致性**：
- 接口需鉴权（基于 token 或会话）；敏感变更需二次授权或审批
- 对写操作使用事务并记录审计日志

**验收标准**：
1. [ ] 提供基础的设备查询/详情/状态变更 API 文档（OpenAPI/Swagger）
2. [ ] 所有写接口产生审计日志，且审计记录包含操作人/时间/前后值

### 5.1.6 可运维性与监控

**功能描述**：
为设备管理模块提供运行时监控与健康检查项（接口延迟、错误率、设备同步失败次数等），并在管理员界面显示关键指标与告警入口。

**验收标准**：
1. [ ] 管理员可查看设备模块的基本监控指标（请求成功率、平均响应时间）
2. [ ] 设备批量导入/同步失败能在界面查看失败明细并支持重试


### 5.2 文档管理模块

#### 5.2.1 文档生命周期管理

**功能描述**：
管理文档从创建到归档的完整生命周期。

**文档状态流转**：
```
起草 → 审批中 → 生效 → 变更中 → 归档/作废
```

**状态说明**：
| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| 起草(Draft) | 文档正在编辑 | 编辑、提交审批、删除 |
| 审批中(Pending) | 等待审批 | 查看、审批通过、审批拒绝 |
| 生效(Active) | 当前有效版本 | 查看、下载、发起变更、发起借阅 |
| 变更中(Changing) | 新版本正在准备 | 查看当前版本、新版本上传 |
| 归档(Archived) | 历史版本，仅供查阅 | 查看、下载 |
| 作废(Deprecated) | 不再使用 | 查看 |

**验收标准**：
1. [ ] 新上传文档默认状态为"起草"
2. [ ] 文档提交审批后状态变为"审批中"
3. [ ] 审批通过后状态变为"生效"
4. [ ] 生效文档被新版本替换后自动归档
5. [ ] 作废文档保留但明确标注，不可下载

#### 5.2.2 版本控制

**功能描述**：
实现文档多版本管理，支持版本对比和历史回溯。

**版本号规则**：
- 格式：`主版本.次版本`（如 1.0, 1.1, 2.0）
- 新建文档版本为 1.0
- 同一设备+同一文档类型的文档，新上传时次版本+1
- 重大变更（如重新验证）可手动设置主版本+1

**版本管理功能**：
- 版本列表：显示所有历史版本及变更说明
- 版本对比：可选两个版本对比差异（文本/图片类型）
- 版本回退：管理员可将生效版本回退到历史版本（需审批）
- 版本下载：可下载任意历史版本

**验收标准**：
1. [ ] 同一设备+文档类型上传新版本时自动递增版本号
2. [ ] 版本列表显示版本号、创建人、创建时间、变更说明
3. [ ] 可下载任意历史版本
4. [ ] 版本变更记录记入审计日志

#### 5.2.3 文档检索

**功能描述**：
提供多维度文档检索能力。

**检索方式**：
| 检索类型 | 说明 |
|----------|------|
| 全文检索 | 检索文档名称、描述、关键词 |
| 高级检索 | 按设备/类型/状态/上传人/日期范围组合检索 |
| 标签检索 | 按自定义标签检索 |
| 智能推荐 | 根据当前设备推荐相关文档 |

**验收标准**：
1. [ ] 支持按文档名称模糊搜索
2. [ ] 支持按设备名称/编码筛选

| 文档版本ID | 被签名的文档版本 |
| 文档哈希 | SHA-256(文档内容+签名信息) |
| IP地址 | 签名时的终端IP |
| 终端信息 | 浏览器/设备信息 |

**验收标准**：

### 5.6 审计追踪模块

#### 5.6.1 操作审计

**功能描述**：
完整记录系统中所有操作，不可篡改。

**审计追踪8个必填字段**：
| 字段 | 说明 | 示例 |
|------|------|------|
| 时间戳 | 操作发生的精确时间 | 2026-05-11 14:30:25 |
| 操作者 | 执行操作的用户 | zhangsan |
| 操作类型 | 操作的类别 | CREATE/UPDATE/DELETE/VIEW/APPROVE/SIGN |
| 操作对象 | 被操作的对象类型和ID | DOC:1001, DEV:203 |
| 操作前值 | 操作前的数据状态 | {"status": "draft"} |
| 操作后值 | 操作后的数据状态 | {"status": "active"} |
| 修改原因 | 操作的原因（用户输入） | 根据验证结果更新状态 |
| IP地址 | 操作终端IP | 192.168.1.100 |

**不可篡改机制**：
1. 审计日志写入WORM存储（一次写入，多次读取）
2. 每条日志包含前一条日志的哈希（哈希链）
3. 时间同步NTP服务器
4. 仅管理员可查看，无编辑/删除权限

**验收标准**：
1. [ ] 所有P0功能操作均记入审计日志
2. [ ] 审计日志不可修改、不可删除
3. [ ] 审计日志可按时间/用户/操作类型/对象筛选
4. [ ] 审计日志可导出（CSV/PDF）
5. [ ] 导出日志包含完整性校验信息

#### 5.6.2 ALCOA++合规

**审计数据符合ALCOA++原则**：

| 原则 | 实现方式 |
|------|----------|
| Attributable（可归因） | 用户账号唯一，操作者不可抵赖 |
| Legible（清晰） | 日志格式规范，易读易懂 |
| Contemporaneous（同步） | 操作即时记录，时间戳精确到秒 |
| Original（原始） | 首次操作即记录，不使用副本 |
| Accurate（准确） | 操作前后值完整记录 |
| Complete（完整） | 所有操作均有记录，无遗漏 |
| Consistent（一致） | 时间格式、记录格式统一 |
| Enduring（持久） | 存储稳定，保存期限≥10年 |
| Available（可用） | 日志随时可查，可导出 |

## 9. 技术架构要求

### 9.1 技术选型

#### 9.1.1 推荐技术栈

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 后端框架 | **Django 4.x** | 1. Django Admin可节省40-60%后台开发工作<br>2. 内置ORM完善<br>3. 内置权限系统完善<br>4. 表单/模板/认证功能完整 |
| 数据库 | **PostgreSQL 14+** | 1. 百万级数据性能远优于SQLite<br>2. 支持JSON类型，便于扩展<br>3. 支持全文检索<br>4. 支持WORM存储特性 |
| 前端框架 | Vue 3 + Element Plus | 1. 组件丰富，开发效率高<br>2. 文档完善<br>3. 与Django REST Framework配合良好 |
| 文件存储 | 本地存储 + 云备份 | MVP阶段使用本地文件系统，定期备份到云存储 |
| 缓存 | Redis | 会话缓存、查询缓存 |
| 搜索 | Elasticsearch | 文档全文检索（可选，P1阶段可先用PostgreSQL全文检索） |

#### 9.1.2 放弃原技术栈的原因

| 原技术栈 | 放弃原因 |
|----------|----------|
| Flask | 微框架，权限/ORM/Admin需自行集成，开发工作量大 |
| SQLite | 1. 单文件数据库，>1GB数据性能急剧下降<br>2. 并发支持弱，>5并发时性能下降<br>3. 无法满足5年180-900GB数据存储需求<br>4. 审计日志高并发写入会锁表 |

| **AI智能功能** | 暂不做智能推荐/OCR/文档摘要 | 后期迭代 |
| **BI高级分析** | 暂不做自定义报表设计器 | MVP提供固定报表 |
| **设备物联网** | 暂不接入设备传感器数据 | 后期迭代 |
| **电子签章** | 暂不支持CA数字证书签章 | 满足电子签名即可 |
| **纸质档案数字化** | 暂不提供扫描OCR入库功能 | 后期迭代 |

### 11.2 范围边界

- 本系统定位为**设备档案文档管理系统**，不涵盖质量管理体系（QMS）的全部功能
- 本系统管理**设备相关文档**，不涵盖生产批次记录、实验记录等
- 本系统为**单机部署版**，暂不提供云原生/容器化部署方案
